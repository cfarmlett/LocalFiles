Status: Completed historical implementation task.

Implement a small V1.5 workflow-validation polish pass for LocalFiles.

Context:

The core V1.5 features are complete.

This task addresses two specific UX issues discovered during manual testing:

1. Split PDF custom ranges can reference pages beyond the document length and currently fail with a generic PDF-processing error.
2. Metadata Removal allows generation of a new PDF even when no removable standard metadata exists, resulting in confusing UX and filenames such as:

```text
budget_receipt-metadata-removed-metadata-removed.pdf
```

This task is intentionally narrow.

Do not redesign workflows or introduce broad validation frameworks.

Goal:

Improve user feedback and workflow behavior when:

- Split custom ranges reference pages that do not exist.
- Metadata Removal detects no removable standard metadata.

Requirements:

## Split PDF

### Problem

Given a 20-page PDF:

```text
1-30
```

currently produces a generic processing failure:

```text
The PDF operation failed. Try again with different PDF files.
```

The PDF is not the problem; the user input is invalid.

### Required Behavior

Validate custom ranges before processing.

If any specified page exceeds the document page count:

- Do not attempt PDF processing.
- Show a user-facing validation error.
- Keep the loaded document intact.
- Do not generate output.
- Do not show the generic PDF-processing failure message.

Example:

```text
Page range exceeds document length.

This PDF contains 20 pages, but the range includes page 30.
```

Exact wording may vary.

### Validation Rules

Any referenced page must be:

- > = 1
- <= document page count

Examples that should be invalid for a 20-page PDF:

```text
1-30
25
10-40
```

Only add page-count validation.

Do not redesign or alter existing range parsing behavior.

Existing valid custom ranges should continue working.

Existing invalid-range handling unrelated to page count should remain unchanged.

### Export Result Requirements

Validation failures must not:

- generate output
- create downloadable results
- create an Export Result Panel

---

## Metadata Removal

### Problem

When no removable standard metadata exists:

```text
No removable standard metadata was found.
```

The workflow still allows generation of a new PDF.

This can produce files such as:

```text
budget_receipt-metadata-removed-metadata-removed.pdf
```

even though no metadata was removed.

### Required Behavior

When no removable standard metadata is detected:

- Clearly communicate that no removable standard metadata was found.
- Keep the loaded file visible.
- Keep the metadata analysis visible.
- Preserve the existing ability to run Metadata Removal if the user chooses to do so.
- Do not block the workflow solely because no removable standard metadata was detected.

The goal is improved clarity, not preventing valid operations.

### Filename Hygiene

If Metadata Removal is run repeatedly:

- Filenames must not accumulate repeated suffixes.
- The filename helper should be idempotent.
- Applying Metadata Removal repeatedly should not continue appending additional `-metadata-removed` suffixes.

Examples:

Valid:

```text
budget_receipt.pdf
→ budget_receipt-metadata-removed.pdf
```

Valid:

```text
budget_receipt-metadata-removed.pdf
→ budget_receipt-metadata-removed.pdf
```

Invalid:

```text
budget_receipt-metadata-removed.pdf
→ budget_receipt-metadata-removed-metadata-removed.pdf
```

### State Requirements

Changing files must:

- clear any previous metadata-analysis state
- clear any prior "no removable metadata" messaging
- analyze the newly selected file normally

### Export Result Requirements

If a previously generated Metadata Removal output exists and the user loads a different file:

- stale output must be invalidated normally
- Export Result Panel behavior must remain consistent with UX-003
- stale downloadable results must not remain visible

---

## Out of Scope

Do not:

- Add validations for merge-one-file workflows.
- Block one-page splits.
- Block one-page reorders.
- Block one-file merges.
- Add warnings for low-value but valid operations.
- Change export-result architecture.
- Change Clear behavior.
- Change Reset Order behavior.
- Change drag-and-drop behavior.
- Introduce telemetry, analytics, accounts, uploads, or dependencies.
- Introduce a generic validation framework.

---

## Architecture Requirements

- Keep validation local to the affected workflows.
- Reuse existing error-display patterns.
- Preserve existing local-only/privacy-first behavior.
- Avoid unrelated refactors.
